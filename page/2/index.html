接口异常测试

异常用例的设计

应用层面

通过分析应用调用关系，然后考虑可能出现的依赖服务的异常

  1. MQ:消息乱序、重复。
  2. 银行：调用银行扣款失败。
  3. KV：读写 KV 失败。
  4. DB：访问 DB 失败。
  5. 其他应用：

系统层面
最后从整体系統考的部署、拓扑、分布式进行分析


需要考虑的场景有：1. 网络故障。2. 机房宕机

异常测试的分类实施

根据上面的例子，可以基本了解到异常测试如何设计。在本节对于每种异常进行详细解释以及给出需要关注的点。

功能异常

功能测试是基于用户使用场景进行设计的异常，是功能测试重要的补充部分，一般建议在功能测试阶段中就进行考虑和执行，一般采用手动的方式执行其中操作异常属于比较典型的功能异常，需要根据对业务的了解设计场景，
拿退款的场景来说，有 1. 重复操作：如一笔订单退款完再退款。2. 乱序操作：如还未支付成功就进行退款。3. 违规操作：如退别人的款、退款超过支付金额限制。4. 操作中断：如未提交刷新/回退页面。

执行过程中需要关注

  1. 报错信息是否恰当、准确。2. 错误操作后功能是否可用。

接口异常

接口异常需要关注接口逻辑的正确性

代码内部逻辑处理是否正常接口对外表现是否符合预期 3. 异常信息返回是否符合要求。

最常见的接口异常的用例就是接口参数检查的用例了。

一般接口参数需要对非法的参数进行检查，同时对于参数的类型和长度也需要做一定限制。非法的参数处理不当容易造成脏数据的问题，导致后续业务处理异常，严重的情况如 SQL 注入还会导致安全方面的问题，需要引起重视。
常见的处理方式有：特殊字符的转义或是报错拦截等。

除了非法参数，调用接口时的常见异常情况还有重复访问、并发访问等，这往往要求接口逻辑里采用事务机制、幂等性、锁等方式进行处理，这些都可以作为验证的点。

  1. 触发某一步骤失败，失败后回滚，无数据残留

  2. 失败后，请求重入能够成功

类别 检查项 检查点
幂等 重复收到请求，业务需要确保操作唯一性 多次操作的结果是一致的，比如脚本重复执行后结果一致
事务 业务要求事务性，在执行过程中需要保证所有步骤都成功才最终成功 1. 触发某一步骤失败，失败后回滚，无数据残留,
2. 失败后，请求重入能够成功
重入 业务在执行过程中某一步出现失败，再次调用 再次调用能够成功
并发 对唯一性资源进行并发请求，如并发执行更新、发放、扣減操作 资源只能操作一次，仅有一个请求能够被正确处理

依赖服务的异常

依赖服务的异常指的是依赖的服务无法正常响应时，观察系统的表现。

主要类型

  1. 服务故障：因为各种原因的无法响应请求。2. 服务报错：返回特定错误、异常。

  2. 状於态不一致：如：异步处理的异常或者底层服务处理成功，上层服务处理响应失败或未收到请求。

说到依赖，会有一个基本概念叫做强依赖和弱依赖：不影响核心业务流程，不影响系统可用性的依赖都可以叫做弱依赖，反之就是强依赖。

系统之间调用是很复杂的，有应用之间的调用，也有对缓存、DB 等中间件的调用，需要思考ー下：锁、鍰存、消息等是不是对业务来说必须的，应该是强依赖还是弱依赖。几个处理依赖服务异常的原则

  1. 除非走投无路，不会因为依赖的系统引起当前应用出现系统级可用性的故障。2. 对于弱依赖，一般都要配置限流或是自动降级策略。3. 理想情況下，应用启动应该做到 0 虽依赖启动。4. 通过本级的报错信息能够知道是下级哪个依赖引起的问题。

系统异常

前面的接口异常以及依赖服务的异常往往关注的是异常对局部功能的影响，而系统异常是在系统实际拓扑架构下，注入异常，观察系统层面的稳定性，需要关注整个系统功能的可用性和系统的故障处理机制。这种方式需要精心设计，以及多方配合进行执行和观察，一般的实施方式是进行线上演练。

项目测试流程

一、需求评审

项目启动之前，需要有需求，并且对需求进行评审。测试同学在参与需求评审之前，需要认真阅读需求文档，从而获取相关的信息

  1. 这个需求是哪些角色使用的。例如是买家还是卖家，是高级管理员还是库管人员。

  2. 不同角色，在什么场景下使用这个功能。例如是 PC 端还是手机端，是微商城版本还是零售版本

  3. 整个项目是否存在对象的生命周期流转，比如订单状态的流转、任务的开始执行到执行完毕，同时需要思考在什么条件下会发生怎样的流转。需要明确状态发生变更的条件规则，确保对象生命周期是有终态的。

  4. 明确每个业务点的人机交互过程和规则，业务过程是否连贯明确

    。在需求评审时，就相关问题和风险与产品和开发进行沟通，直到问题得到明确答复

    二、技术方案评审

需求评审后，开发将技术方案文档准备好，可以开始进行技术方案评审。在技术方案评审中，对于架构、数据表等技术方案细节需要做充分的评估。测试人员需要了解正常的业务流程和实现机制，是否能够实现需求中描述的功能，对其合理性进行评估。此外，还需要关注技术方案中

**$\color{#FF3030}{并发的控制、幂等的控制、超时的影响、缓存的使用、数据库表索引和性能、事务控制以及异常的处理}$**
```。
这些问题在正向流程中往往很难被发现，但是到了线上以后，由于调用量巨大、场景复杂，往往会造成较严重的问题。

由于有赞整体系统业务和模块众多，应用之间调用关系较复杂，还需要注意评估：

・项目对外部业务影响

项目对上层业务影响

数据模型变更和数据迁移的方案

配套的检查和核对程序，比如库存核对、账务核对

### 三、项目立项

项目立项时，开发和测试人员需要根据对目前需求和技术方案的了解，拆分研发任务，评估时间，输出几个重要的里程碑事件点：开发截止时间、测试方案和用例评审时间，测试完成截止时间、上线时间。

有时候会遇到项目太大，评估的时间太长的情况，可考虑进行子项目拆分

对于无强耦合的业务线可单独拆分出子项目先行测试和发布，一般建议子项目的研发周期在 1 个月以内。

对于强耦合的业务，可考虑先对开发完成的功能进行接口测试，再进一步集成。

### 四、测试方案评审

由于很多大型项目涉及多个业务方的测试资源，測试周期和轮次较长，测试负责人需要准备好测试方案，召集各业务线测试负责人进行测试方案的评审最基本工作是对当前功能进行罗列，划分到对应的测试人员进行覆盖，避免出现归属不明导致漏覆盖的情况。

在明确了各业务线测试人员的测试范围后，确认下安排几轮測试以及各业务线的投入时间和完成时间，从而推断出整体的測试时长和完成。一般考虑应用是否是核心应用，开发改动量如果出现投入时间参差不齐的情况，可以考忠是否拆分批次进行测试，最后再安排集成测试的轮次。

除了测试执行范围和时间的评估，对于测试环境和测试数据的淮备工作也需要确定下来，在项目提测前完成。

### 五、测试用例评审

需求和方案评审完成后，测试可以开始编写测试用例，然后邀请开发和产品一起进行对用例进行评审，记录下评审中发现的问题。

评审结束后需要补充遗漏的点，同时标注下冒烟用例，给出大概的比例；给出冒烟用例提供的时间点，不耽误开发自测。

### 六、冒烟测试

完成代码开发和联调后，开发会进行提测，测试根据提测文档接受提测后，开始进行冒烟测试，主要针对主流程的场景进行测试，查看看是否有功能未完成或者阻塞后续测试的情况。冒烟测试结束后，群里同步冒烟测试结果。

冒烟用例需要 100%通过；冒烟通过后再进行第一轮新功能测试，不通过打回

### 七、功能测试

在功能測试阶段，发现问题由对应的测试同学提交 bug，在功能测试前需要约定 bug 修复时间，如果开发解决不及时可及时上升给 TL。

业务线测试负责人在功能测试阶段**定期反馈进度**，测试 ptm 汇总每日的项目测试报告发邮件或发送到项目群里。

根据测试策略分成几个轮次进行测试，一股会分为两轮

* 第一轮新功能测试。直接在冒烟测试通过时部署的环境完成第一轮测试，不再重新部署。如果第一轮结束质量 ok 且环境可用于验证，可以开始考虑产品和设计师介入验收。当致命、严重 bug 解決率达到 100%，可以进行下一轮测试
* 第二轮回归验证，回归新的改动对原有功能的影响。当 bug 已收敛（每日新增 bug 数则体递减；每日遗留 bug 数整体递减），可以准备到预发环境进行上线前的测试。比较大型的项目，新功能的测试会分为两轮或者更多轮，最后一轮为项目集成测试，在项目集成測试中，需要对业务线完整的流程进行集成测试，需要注意几点

1. 如果是大项目拆子项目，在这个环节统一部署一个环境，各业务线走一次全链路回归
2. 回归新的改动对原有功能的影响

3. 測试期间同样锁定环境，部署频率建议控制在半天或一天合入一次

4. 集成测试通过标准：致命、严重 bug 解決率 100%，普通和微小 bug 解決率 70%。达到通过标准才允许进行预发环境的验证
5. 不允许非UI类、文案类的需求变更合入。此时的需求变更，开发和测试同学比较难以评估影响和验证完成，容易影响原有功能，造成漏侧

### 八、产品验收

当測试进入尾声，产品功能基本稳定后，邀请产品对已实现的功能验收完成，确认最终实现效果是否符合预期、确认整体交互流程是否 ok。

### 九、发布前准备

发布前，pm、产品、开发、測试一起讨论上线前的准备，測试需要确认应用都达到可以发布的状态，需要增加的配置、需要变更的 DL 都已经记录在发布方案里面，如果有依赖的应用和顼目需要确认发布的顺序，最后各方确认无误输出一份发布计划，需要包含每个应用的发布顺序、责任人、发布后验证点以及项目回滚方案（回滚顺序、数据修复方案）

### 十、预发回归测试

测试环境验证结束，測试会在预发环境进行回归，主要工作是小的 bugfix 的验证以及基本用例的回归，一般情况下不允许有非 bugfix 类的代码合入，如果出现需求变更导致的代码合入，需要在测试环境充分验证后再进行预发验证。

### 十一、上线

预发验证完成后，确认下上线前的准备工作是否完成

1. 测试工作都完成了，剩余未解決 BUG 的缺陷率《5 分（缺陷率=致命数目*10+严重数目*3+普通数目*1+微小数目*0.1) 
2. 输出项目质量报告，反馈项目质量状况和风险
3. 产品视觉验收通过
4. 输出了发布上线计划
确认完成后可以按发布计划发布，发布完成后测试验证功能，开发和测试发布当天和第二天观察流量和告警，如有功能问题第一时间回液。
最后，当发布验证通过后，同步项目进度到项目群。

### 十二、总结和复盘

项目上线后不代表所有工作都已经做完，首先对项目过程中的质量问题进行分析，主要对于 bug 数目、严重程度和原因进行分析，思考下如何预防和减少类似质量问题。如果是大型项目或者项目的进度或者质量方面存在一些问题，会对项目过程进行必要的复盘，提出一些改进措施和建议，进行针对性改进，使得后续项目能够运行得更顺利。
最后，测试同学需要把项目里新增特性的用例需要加入到基线用例，避免特性的遗漏，同时新增的自动化用例随着项目上线后也需要合入到测试工程的 master 分支上，并确保新增用例和老用例都能运行通过。

需求分析及测试方案设计

一、什么是需求分析？

需求一般来自客户或者开发团队内部，大类别为产品需求和技术改造需求。无论来源，QA 同学必须要求拿到产品 prd 和开发的技术方案。关键的问题一定要落入文档并及时更新。有赞大部分项目无法独占开发资源，项目中途常会发生项目成员更替的情形，此时口口相传的模式总是会出现信息遗失和断层。当然，如果能让产品负责人和技术负责人重新过一遍方案更好，然而在实际项目中此举极少。新的成员只能从头做起，从零碎未整理的对话、群聊天、会议记录等挖取信息，这样的需求理解完整性不足。所以，请保持关键问题落档并及时更新

二、为什么要做需求分析？

前面举过极端例子，有的项目在了解甲方需求之后，可以用非常少的资源（包括时间和人カ）实现。这里并非指摘产品同学专业性，而是在快速发展的互联网公司，一大部分需求在用户量没有起来之前花大力气做的系统完整可靠，投入产出极低，甚至有可能不需要继续迭代。此时，需要 QA 人员能够从商业价值、技术成本等方面综合衡量，给产品和技术负责人提供实现需求的新思路

因此，如果没有做需求分析，可能会有如下的一些不良后果：
1、浪费资源实现用户不常用的需求，供过于求；
2、遗漏了 prd 中没有提到，但是很重要的需求，如漏分销、精选等模块（不是所有产品同学都有全站业务的视图视角，产出的 prd 完整性不足）;
3、需求分析不到位，测试估时不足，导致项目延期或者是强行上线降低线上质量。

三、如何做需求分析？

3.1 通读 prd、技术文档、产品交互稿进行推演

通读产品文档，了解这个需求要实现什么，从而确认测试什么-what，怎么測-how，多长时间-when，多少人测-who，在什么环境测试- where 等。有了整体了解之后，将文档中不确认的点记录下来，形成清单请产品同学确认并落档。清晰的记得某项目中给产品的确认清单列了 50+，干万不要不好意思去找人确认。其一是前期考虑越充分，后期返工的概率越低；其二是产品同学在很多细节上面的考虑由于各种原因确实考虑不足，清单是对产品完整性的一个很好的补充；

研读技术文档也是非常重要的环节，开发的改动大小直接关系着测试工作量。同时，某一模块开发的改动往往关联其他模块功能，这也是非常容易遗漏的地方。有一定经验的测试同学需要在满足产品需求的同时，同时评估开发改动成本和系統拓展性等等，尽可能以较小的代价满足。

3.2 确定测试范围

大部分面向用户的产品都有一定的业务流程，及用户使用该功能实现自己业务的流程。基本的大类如下:

  1. 常用的/必须的主干流程;
  2. 各分支流程的遍历;
  3. 明确不可用的分支流程;
  4. 没有明确规定但是应该不支持的业务流程；
  5. 其他不符合的用户操作
    然后根据产品需求，一项一项列出需要的测试场景，和该场景应该导致的结果，形成基本的测试用例。我们目前只要只用 xmind 梳理思路。

3.3 确定优先级和测试方法

需求的优先级别，是測试人员、产品、开发三者共同确认，产品项目中最为核心的功能。由此可以圈定出部分冒烟測试。同时，在项目进程尤其是测试进度发生堵塞时，进行不同模块、系统、功能的迭代或者取舍，不至于项目停滞。测试方法，如大流量接口的性能测试、支付接口的幂等、L1 接口的白盒測试等等，测试数据是否需要提前准备，三方沙箱环境如何联调，消息处理是否需要隔离等等，都需要在测试方案中提前想好应对措施。

3.4QA 人员介入时间

根据历史经验，跨团队的复杂项目每个模块的交付时间有先有后，PTM（项目测试负责人）和各业务模块的测试人员可以确认哪些模块的测试可以提前介入，比如后端服务的测试可以在项目整体联调之后开始接口自动化测试。这有助于项目进度和提前暴露未估计到的风险

3.5 查漏补缺

在团队内部进行测试方案分享，借助其他同学经验发现个人没有考虑到的点；在项目组内部及时和成员沟通，跟进每日晨会记录，完整和更新用例；关注同模块并行项目，它的改动点对本项目的影响评估等等。

四、如何提高需求分析能力

4.1 梳理业务，熟悉系统

QA 人员在脑海中构建完整系统拓扑图，同时研读代码和了解业务知识，以用户使用者角度熱悉使用产品，以技术人员角色熱悉系统交互和功能逻辑。如果这一块系统交互不太有把握，推荐使用精准测试工具的接口分析

4.2 用户第一位

QA 人员在拿到需求的第一时间首先需要站在用户的角度分析用户需要的是什么，这个是不是通用需求。从商业价值和投入产出上考虑，这个的优先级在什么位置。其次オ是站在开发或者业务实现角度，给产品和开发同学设计实现上的一些建议。虽然后者看起来更为高大上，但脱离前者，事倍功半。

4.3 基线用例与个人思考

每条业务线都有自己的基线用例，对于大部分的项目，基线用例基本够用，好像已经不需要 QA 人员的主动思考了。这里我不太认同。首先基线用例是否实时更新；其次回归所有基线是非常低效且不动脑筋的方式，每一次改动的影响范围和基线用例必然是有交集而非完全包含，全部回归是最傻最低效的方式（自动化全部覆盖为另一说）；第三，漏上线问题的用例极有可能需要补充进基线用例，没有自我思考的 QA 人员漏測概率很大。

4.4 探索性测试与总结分享

集团队众人力量，分享典型性问题，和一些容易踩坑的点，不断提高自己的业务能力水平。保持好奇心，进行探索性测试，不拘泥与手头所知内容。

Linux下防火墙的开启及配置

Posted on 2021-07-24 | In Linux

查看防火墙是否开启

systemctl status firewalld

开启防火墙

systemctl start firewalld

关闭防火墙

systemctl stop firewalld

查看所有开启的端口

firewall-cmd –list-ports

防火墙开启端口访问

firewall-cmd –zone=public –add-port=80/tcp –permanent

命令含义： –zone #作用域 –add-port=80/tcp #添加端口，格式为：端口/通讯协议 –permanent #永久生效，没有此参数重启后失效

添加完后记得重启防火墙 firewall-cmd –reload

三、 通过docker

  1. 下载Jenkins镜像文件 docker pull jenkins/jenkins:lts

  2. 创建Jenkins挂载目录并授权权限（我们在服务器上先创建一个jenkins工作目录 /var/jenkins_mount，赋予相应权限，稍后我们将jenkins容器目录挂载到这个目录上，这样我们就可以很方便地对容器内的配置文件进行修改。 如果我们不这样做，那么如果需要修改容器配置文件，将会有点麻烦，因为虽然我们可以使用docker exec -it –user root 容器id /bin/bash 命令进入容器目录，但是连简单的 vi命令都不能使用）

    • chmod 777 /var/jenkins_mount

  3. 创建并启动Jenkins容器

    docker run -d -p 10240:8080 -p 10241:50000 -v /Users/admin/docker/jenkins_mount:/var/jenkins_home -v /etc/localtime:/etc/localtime –name myjenkins jenkins/jenkins:lts

  4. 启动成功后，查看docer容器日志

    docker logs myjenkins
